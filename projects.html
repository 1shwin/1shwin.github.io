What I'd do differently now:

termSync

What it does: A real-time, collaborative rich text editor designed for the terminal. Allows multiple users to connect to a central server to edit a document simultaneously. It features live cursor tracking, text formatting, and an integrated C code execution environment.

Why it was built: Built for Megathon '25 @ Infinium (in < 24 hours)

What I'd do differently now:

  • Code execution currently runs directly on the host server. I'd add containerization to maintain security and stability.
  • Add an advanced authentication system with different levels of permissions.

Technology tradeoffs:

  • Used unencrypted TCP/IP prioritizing low-latency updates over out-of-the-box security for use in trusted local networks, instead of adding TLS.

Link: termSync

TwixT

What it does: A terminal-based implementation of the classic 2-player strategy board game. It allows players to place pegs on a grid and connect them using knight's move links. The program manages the board state, renders the grid and links, validates legal moves and detects winners.

Why it was built: Built as a course project for CS1.101 Computer Programming.

What I'd do differently now:

  • Move the game engine (link validation and win-checking) into a standalone library so it could be easily ported.
  • Use a more modern library like ncurses for better window management and resizing support compared to standard C terminal print.

Technology tradeoffs:

  • Using vanilla C gave the program high portability at the cost of advanced TUI elements from a modern library.

Link: TwixT
